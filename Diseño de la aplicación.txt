Diseño de la aplicación local: DevControl Local
1. Concepto general

DevControl Local sería una aplicación instalada en la computadora del usuario para controlar proyectos desarrollados con IA antes, durante y después del uso de un editor agéntico como Cursor, Claude Code, GitHub Copilot, Windsurf, VS Code u otros.

No debe comportarse como otro editor de código. Su función debe ser:

Inicializar, proteger, auditar y controlar el proyecto antes de que el agente IA pueda modificar archivos.

La aplicación debe arrancar antes del primer prompt, cargar las reglas del proyecto, conectar con el repositorio, validar el stack, proteger archivos críticos y registrar toda interacción técnica relevante.

2. Problema que resuelve

La mayoría de editores agénticos empiezan desde el prompt:

Usuario → Prompt → Agente IA → Cambios en archivos

Ese flujo es débil porque el control llega tarde.

El flujo correcto debería ser:

DevControl Local → Validación del proyecto → Reglas activas → Editor agéntico → Prompt → Plan → Validación → Cambio controlado

La aplicación debe actuar como sistema de precontrol.

3. Objetivo técnico

La aplicación debe garantizar que, antes de escribir el primer prompt:

el proyecto tenga estructura reconocida;
exista control de versiones;
haya reglas para agentes;
se definan archivos protegidos;
se registre el stack autorizado;
se limite el alcance de los cambios;
se preparen hooks de validación;
se active monitoreo de archivos;
se bloquee la eliminación peligrosa;
se cree una memoria técnica del proyecto;
se conecte con el editor o herramienta agéntica;
se genere un registro de auditoría.

6. Punto crítico: arrancar antes del primer prompt

Para lograrlo de forma realista, no basta con una app abierta. Se necesita un mecanismo de arranque controlado.

Flujo recomendado
1. Usuario abre DevControl Local
2. Selecciona o crea proyecto
3. DevControl valida estructura
4. DevControl instala reglas y hooks
5. DevControl inicia monitoreo local
6. DevControl lanza el editor agéntico
7. El editor abre el proyecto ya protegido
8. El usuario recién puede enviar prompts

El editor no debería abrirse directamente desde el acceso normal. Debe abrirse desde DevControl:

devcontrol launch cursor
devcontrol launch claude
devcontrol launch vscode

Esto permite asegurar que el proyecto ya esté bajo control antes de que el agente actúe.

7. Advertencia técnica importante

No todos los editores permiten interceptar directamente el texto del prompt antes de enviarlo al modelo.

Por eso, la aplicación debe controlar cuatro puntos posibles:

Nivel de control	Qué controla	Viabilidad
Antes del editor	Valida proyecto, reglas y permisos	Alta
Dentro del editor	Extensión, reglas, instrucciones del agente	Media
Durante ejecución	Monitoreo de archivos y comandos	Alta
Después del cambio	Git diff, tests, auditoría, aprobación	Alta

El control total del prompt solo sería posible si:

el editor permite extensión oficial;
el agente usa MCP;
el usuario usa una CLI controlada;
el tráfico pasa por un gateway propio;
o DevControl ofrece su propio panel de prompts.

Por eso el diseño correcto debe ser híbrido.

Módulo 2: Motor de políticas

Este es el núcleo de la aplicación.

Debe decidir qué puede o no puede hacer el agente.

Ejemplo de niveles:

Nivel	Acción	Permiso
Verde	Cambiar textos visibles, estilos menores, documentación	Permitido
Amarillo	Crear componentes, endpoints o servicios nuevos	Requiere plan
Naranja	Modificar dependencias, estructura o modelos de datos	Requiere aprobación
Rojo	Eliminar archivos, tocar auth, pagos, base de datos o producción	Bloqueado
Negro	Ejecutar comandos destructivos, borrar datos o cambiar producción	Prohibido
Módulo 3: Control de archivos protegidos

La aplicación debe permitir definir archivos y carpetas que el agente no puede modificar sin autorización.

Ejemplo:

{
  "protectedFiles": [
    "package.json",
    "package-lock.json",
    "pnpm-lock.yaml",
    ".env",
    ".env.local",
    "prisma/schema.prisma",
    "src/auth/**",
    "src/payments/**",
    "src/database/**",
    "src/config/**",
    "docker-compose.yml",
    "Dockerfile",
    ".github/workflows/**"
  ],
  "protectedActions": [
    "delete_file",
    "rename_file",
    "modify_dependencies",
    "run_migration",
    "change_auth_logic",
    "change_payment_logic",
    "modify_env",
    "push_to_main"
  ]
}

Regla clave:

Ningún agente puede eliminar, renombrar o sobrescribir archivos protegidos sin aprobación explícita.

Módulo 5: Control de comandos

Los agentes agénticos pueden ejecutar comandos en terminal. Esto es peligroso.

DevControl debe bloquear o pedir aprobación para comandos como:

rm -rf
del /s
git reset --hard
git clean -fd
npm uninstall
pnpm remove
npm install
pnpm add
prisma migrate
docker compose down -v
drop database
truncate table
git push --force

Clasificación:

Comando	Riesgo	Acción
npm run lint	Bajo	Permitir
npm test	Bajo	Permitir
npm install	Medio	Pedir aprobación
prisma migrate dev	Alto	Pedir aprobación reforzada
rm -rf	Crítico	Bloquear
git push --force	Crítico	Bloquear
docker compose down -v	Crítico	Bloquear

11. Integración con editores agénticos
A. Cursor

DevControl debe generar:

.cursor/rules/devcontrol-agent-rules.mdc

Contenido base:

# DevControl Agent Rules

You are working inside a protected software project.

Mandatory rules:

1. Do not delete files unless the user explicitly approves.
2. Do not modify package files unless the task requires it and approval is given.
3. Do not change the selected stack.
4. Do not modify authentication, payments, database schemas, environment files or deployment files without approval.
5. Before editing more than 3 files, present a plan.
6. Prefer small, focused changes.
7. Never rewrite complete files when a partial change is enough.
8. After every change, summarize:
   - files modified;
   - reason for change;
   - risk level;
   - tests needed.
9. Follow the architecture in `/docs/02-architecture.md`.
10. Follow the UI system in `/docs/04-ui-system.md`.

Trabajarás bajo las siguientes restricciones:

- `pnpm-lock.yaml`
- `.env`
- `.env.local`
- `src/auth/**`
- `prisma/schema.prisma`
- `.github/workflows/**`

## Reglas
1. No elimines archivos.
2. No cambies dependencias.
3. No modifiques configuración global.
4. No cambies autenticación.
5. No cambies base de datos.
6. No reescribas archivos completos si puedes hacer cambios parciales.
7. Antes de modificar archivos, presenta un plan.
8. Indica los archivos que tocarás.
9. Espera aprobación antes de ejecutar cambios.
10. Al finalizar, entrega resumen técnico y pruebas recomendadas.

## Documentos obligatorios
Lee primero:

- `/docs/00-project-brief.md`
- `/docs/02-architecture.md`
- `/docs/04-ui-system.md`
- `/docs/07-agent-rules.md`

17. Control de contexto

La aplicación debe tener una memoria local del proyecto, no una memoria dependiente del chat.

Archivos de contexto
.devcontrol/context/
├── project-summary.md
├── current-sprint.md
├── active-decisions.md
├── forbidden-changes.md
├── known-errors.md
├── completed-modules.md
└── pending-tasks.md

Antes de cada sesión, DevControl debe generar un resumen compacto:

# Contexto activo para agente

Proyecto:
AutoMantPro

Estado actual:
Módulo de vehículos completo.
Módulo de diagnóstico en desarrollo.

No modificar:
- autenticación
- estructura de base de datos
- package.json

Decisiones vigentes:
- UI con Tailwind + shadcn/ui
- formularios con react-hook-form
- validaciones con Zod
- rutas bajo app router

Errores conocidos:
- No duplicar VehicleForm
- No crear otro cliente de Supabase
- No mover componentes globales

Esto reduce pérdida de contexto y consumo de tokens.

18. Control de costos de tokens

Para editores cerrados, DevControl no siempre podrá medir tokens exactos. Pero sí puede controlar comportamiento.

Debe registrar:

duración de sesión;
número de archivos leídos;
número de archivos modificados;
prompts enviados desde panel propio;
tamaño del contexto;
número de iteraciones;
modelo usado, si está disponible;
costo estimado, si el proveedor lo permite.

Reglas de eficiencia:

[✓] No usar modo agente para cambios menores.
[✓] No cargar todo el repositorio si la tarea afecta un módulo.
[✓] No permitir prompts sin alcance.
[✓] No permitir modificaciones en más de 5 archivos sin plan.
[✓] No permitir refactorizaciones globales sin tarea formal.
19. Control de Git obligatorio

DevControl debe exigir Git activo.

Si el proyecto no tiene Git:

Este proyecto no tiene control de versiones.

No se recomienda usar un agente IA sin Git.

Opciones:
[Inicializar Git] [Cancelar]

Antes de iniciar sesión IA:

git status

Si hay cambios sin guardar:

Hay cambios sin commit antes de iniciar la sesión.

Opciones:
[Crear snapshot] [Crear commit] [Continuar sin protección — no recomendado]

Regla fuerte:

No se debe iniciar una sesión agéntica si no existe posibilidad de rollback.

eliminación masiva;
modificación de archivos protegidos;
secretos;
cambios en lockfiles sin aprobación;
cambios en .env;
commits directos a main;
commits sin mensaje estructurado.

Ejemplo:

devcontrol check --pre-commit

27. Reglas obligatorias que debe aplicar

Estas son las reglas que debe tener desde la primera versión.

Reglas de proyecto
1. Todo proyecto debe tener Git activo.
2. Todo proyecto debe tener snapshot inicial.
3. Todo proyecto debe tener stack autorizado.
4. Todo proyecto debe tener reglas del agente.
5. Todo proyecto debe tener archivos protegidos.
6. Todo proyecto debe tener documentación mínima.
Reglas del agente
1. No eliminar archivos sin aprobación.
2. No modificar dependencias sin aprobación.
3. No cambiar arquitectura sin aprobación.
4. No modificar auth, pagos, base de datos ni deploy sin aprobación.
5. No ejecutar comandos destructivos.
6. No reescribir archivos completos sin justificar.
7. No tocar más de 3 archivos sin plan.
8. No introducir librerías fuera del stack.
9. No crear UI genérica fuera del sistema visual.
10. No finalizar sin resumen técnico.
Reglas de seguridad
1. Bloquear `.env`.
2. Bloquear secretos en commits.
3. Bloquear comandos destructivos.
4. Bloquear pushes forzados.
5. Bloquear cambios en producción.
6. Validar dependencias nuevas.
7. Detectar credenciales expuestas.
Reglas de trazabilidad
1. Todo cambio debe tener sesión asociada.
2. Toda sesión debe tener objetivo.
3. Todo cambio crítico debe tener aprobación.
4. Todo rollback debe quedar registrado.
5. Todo prompt importante debe quedar resumido.
6. Todo cambio aprobado debe actualizar changelog.
28. Diferenciador del producto

La mayoría de herramientas actuales ayudan a escribir código, revisar PRs o detectar vulnerabilidades.

DevControl Local no compite con ellas. Las coordina.

Su propuesta sería:

Antes de usar IA para programar, DevControl prepara, protege y gobierna tu proyecto para evitar cambios no solicitados, pérdida de contexto, eliminación de archivos, mala arquitectura, interfaces genéricas y costos innecesarios.

31. Definición final del producto

DevControl Local es una aplicación de instalación local que se ejecuta antes del editor agéntico y controla el entorno de desarrollo con IA mediante reglas, permisos, archivos protegidos, monitoreo de cambios, Git hooks, auditoría, contexto técnico y validaciones automáticas.

Su principio central:

La IA puede asistir en el desarrollo, pero no debe operar sin arquitectura, límites, trazabilidad ni aprobación.

Ese es el enfoque correcto. No es otro editor. Es la capa de control técnico para programar con IA de forma segura y profesional.
